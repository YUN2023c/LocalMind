> 你的知识，应该躺在你的硬盘里——而不是某个你够不着的服务器上。

这是一款一个人写的、跑在你自己电脑上的本地知识管理应用。

它不收订阅费。

不把你的文档上传到云端。

也不假装自己是企业级解决方案。

它只想安静地帮你把散落各处的文档收拢、整理，并随时可以被你提问。

***

## 一、一句话定义与核心受众

**一句话定义：** LocalMind 是一款本地优先的桌面知识管理应用——它不是云端 SaaS，不向你按月收费，也不把你的文档同步到任何你看不见的服务器。它只是一个跑在你自己电脑上、数据只躺在本地 SQLite 里的知识管家。

**它服务的人很具体，不是"所有人"：**

- 受够了云端笔记每次打开都要转圈"加载中..."、却还要按月付订阅费的独立开发者；
- 笔记里有未发表论文、机密代码、私人手稿，对云端同步有合理隐私焦虑的研究人员；
- 硬盘里堆了几百篇 PDF 论文，但每次想找"那句话到底出自哪一篇"都要翻半天的研究生；
- 想要一个真正属于自己的"第二大脑"，但不想把思考过程交给大厂的文字创作者。

如果你符合上面任何一条——LocalMind 大概率就是为你做的。

如果你要的是多人协作的团队知识库——那它目前真的不是。这一点我会在第三章诚实说明。

***

## 二、核心功能（2-3个）

我不打算堆砌技术参数。

下面三个功能，每一个都对应一个你自己大概也遇到过的麻烦。

### 1. 本地优先的文档管家 —— 解决"文档散落各处、云端又有隐私顾虑"

你只要指定一个学习资料文件夹，LocalMind 就会自动扫描入库里面的 PDF / Word（doc/docx）/ Markdown / TXT / HTML。

它不会一次性把你硬盘翻个底朝天。

而是基于文件修改时间与大小做增量同步——你新加的、改过的才入库，剩下的它不打扰你。

最关键的一点：所有内容只躺在你的本地 SQLite 数据库里。

没有后台悄悄上传。

没有"为了给你提供更好服务而收集你的数据"。

拔掉网线，它照常工作。

### 2. 双模式 AI 问答与摘要 —— 解决"想问文档问题，又怕内容泄露"

这是我最纠结也最得意的一块。

我给它做了两种推理模式，让你自己选：

- **远程 API 模式：** 你自带 API Key（Key 用 keytar 存进系统钥匙串，不是明文躺在配置文件里），调用你信任的远端模型。效果强，但需要联网。
- **本地模型模式：** 一键下载内置的 Qwen2.5-1.5B-Instruct（q4_k_m 量化，约 1GB），断网也能跑。隐私拉满，效果对一个 1.5B 的小模型来说已经相当能打。

问答时它会**流式作答**，你可以随时**停止生成**；每次回答会**标注引用来源**，告诉你这次话出自哪篇文档；问答历史会被保存下来；还有 Token 节省模式给那些按量计费的人。

另外，任何一篇文档都能**一键生成摘要**。

### 3. 强大的检索与管理功能 —— 解决"文档找不到"的问题

文档多了之后，真正折磨人的是"找不到"。

LocalMind 给了三件套：

- **SQLite FTS 全文检索**（带中文分词回退方案）——秒级搜到那句话；
- **标签云 / 关键词 / 收藏夹 / 最近文档**——从不同维度把文档捞出来；
- **文档预览与编辑**——PDF / Markdown / Word 直接看，Markdown 还能改。

> 知识管理的关键，是"想用时能不能立刻找到"。

## 三、灵感来源与痛点狙击（The "Why"）

### 一个真实的崩溃瞬间

一个周末，我坐在电脑前整理错题。

桌面上散落着各科的 PDF 试卷、用 Markdown 记的错题本、从知乎和公众号另存下来的知识点总结，还有自己从网上论坛下载的讲义和网课视频配套笔记。

我记得之前看到过一道物理题的解题思路——但翻遍了"高三复习"文件夹里的二十多个子文件夹，还是找不到。

我花了将近四十分钟。

用文件搜索工具一个一个找。

用 PDF 阅读器一篇一篇 Ctrl+F。

用记事本打开 Markdown 一页一页翻。

最后终于找到了——但那天下午的复习计划全被打乱了。

更憋屈的是后面：我想把这些资料放进某个云端笔记工具，让 AI 帮我归纳考点、自动生成错题解析——但一打开就是各种广告弹窗和 VIP 付费界面，根本没法专心用。

于是我只能继续手动分类。整理到一半，我开始怀疑人生。

### 那个"如果……就好了"的瞬间

就在那个下午，我脑子里冒出一句话：

> 如果有一个工具，能让我的文档不出本地，又能像 ChatGPT 一样被随时提问、被随时检索就好了。

市面上不是没有本地知识库工具。

但要么太重。

要么要自己搭一整套复杂环境。

要么只支持单一格式。

作为一个会写点代码的学生，我决定自己写一个——轻一点、诚实一点、只服务我自己这种场景就好。

### 判断与取舍（这部分我必须诚实）

作为一个人在做的项目，清醒地"砍"比兴奋地"加"重要得多。

下面这些是我主动砍掉或没做的，以及为什么：

- **砍掉了云端同步。** 个人维护一台服务器，成本先不说，一旦数据上了云，隐私风险就不再可控。这违背我做这个项目的初衷。
- **砍掉了协作功能（高亮/笔记/评论共享）。** 单人场景下完全够用；一旦做协作，权限管理、数据一致性、冲突合并的复杂度会成倍上升，一个人扛不动，也容易把产品做歪。
- **没有上向量数据库，用的是 SQLite FTS。** 这一点我甚至有点骄傲地承认。向量库对个人规模的文档库来说太重了，还要额外起一个服务。SQLite FTS 零额外依赖、跟着主进程一起启动，对几百到几千篇文档的检索完全够用。
- **目前只做了 Windows。** 跨平台要测三端、要处理三套打包和原生依赖，一个人真的扛不动。Mac/Linux 是我想做的——但不是现在。

> 独立开发者最大的优势不是"什么都能做"——而是"敢于不做什么"。

## 四、创新性与实用性（The "What's New"）

我不会说"行业首创"这种话——那太像营销稿了。

我想说的是：**在"本地隐私知识库"这个具体场景下，我换了一种更轻量的解法。**

### 个人开发者的优势，在这类场景里反而成了护城河

- **无历史包袱。** 不用背着老版本的兼容债，架构可以从第一行就为"本地优先"设计。
- **决策极快。** 想到一个改进，当晚就能上线；不用走产品评审、不用排期。
- **对边缘需求极度敏感。** 大厂产品不会为了"断网也能跑"而内置一个 1GB 的小模型——但我会，因为我自己就经常在高铁上写东西。

### 几个我觉得值得讲讲的取舍

1. **双模式推理：把"效果 vs 隐私/离线"的选择权交还给你。**
   远程 API 模式给你最强效果，本地模型模式给你最强隐私和离线能力。你不一定要二选一——敏感文档走本地，公开资料走远程，这是只有"双模式"才给得起的灵活度。
2. **SQLite FTS 而非重型向量库：换来零依赖、秒级启动。**
   你不用装 Docker、不用跑一个额外的向量服务、不用担心它崩了要重启。整个应用就是一个 Electron 包，双击即开，关掉即走。
3. **keytar 把 API Key 放进系统钥匙串，而不是明文存盘。**
   这是件小事，但很多本地工具没做。你的 Key 不会以明文躺在某个 JSON 里被你一不小心就提交到 Git。
4. **任务队列（并发控制 + 限流）。**
   一次性拖进去几百篇文档不会把机器卡死，它会老老实实排队、限流、一篇一篇来。

> 诚实地说：LocalMind 不是最强大的——但它可能是最"不打扰你"的那一个。

## 五、应用前景与未来推演（The "Vision"）

我会尽量把话说大——但每一句都得逻辑自洽、不画大饼。

### 短期：解决当下具体的麻烦

这就是 LocalMind 现在正在做的事，而且已经能做：

- 把散落在硬盘各处的文档收进一个可检索的本地库；
- 在不泄露内容的前提下，让 AI 帮你问答和摘要；
- 用知识图谱和全文检索，把"找不到"和"看不出联系"这两个老问题解决掉。

这个层面，它就是一个好用、安静、诚实的本地知识管家。

### 中期（未来构想）：从"知识库"探索为"个人第二大脑的本地操作系统"

如果有一天，LocalMind 加上插件机制和一个小社区——它**有望**不再只是一个本地笔记/知识库，而**探索方向**是成为"个人第二大脑的本地操作系统"——你可以往里面挂自己写的小工具：自动抓取订阅源、定期摘要、和本地的日历/待办联动。所有这些，数据都不出本地。

> 我用"有望"和"探索方向"——是因为这些还只是我脑子里的草图，不是承诺。

### 长期（未来构想）：一个数据不出本地的个人 AI 知识中枢

把视线再拉远一点。

在 AI Agent 和隐私计算的大趋势下，"个人数据留在本地、按需被本地或可信模型调用"这件事，会越来越重要。

LocalMind 的**潜在场景**，不仅仅局限于个人用户：

- **企业保密场景**：那些有严格数据安全要求的企业，或许可以把 LocalMind 作为员工的个人知识库工具——文档留在员工本地，AI 能力可以对接企业内部部署的模型，既满足安全合规，又不影响个人效率。
- **研究所与学术团队**：科研机构往往有大量未公开的实验数据和论文草稿，这些内容绝不能上传到公有云。LocalMind 的本地优先特性，**有望**成为科研人员管理敏感资料的轻量选择。

而对个人来说，它的生态位依然是：

> 一个数据不出本地的个人 AI 知识中枢——你所有的文档、笔记、问答历史都在你自己的硬盘上，而 AI 能力（无论是本地小模型还是你授权的远程 API）围绕你的数据来服务你，而不是反过来。

我必须再次强调：以上中、长期内容均为**未来构想**，**尚未实现**。我不希望任何人因为我对未来的热情，而对它的现状产生误判。

## 技术栈

诚实标注，没有粉饰：

- **Electron** `^42.4.1` —— 桌面应用外壳
- **node-llama-cpp** `^3.18.1` —— 基于 llama.cpp 的本地推理（驱动 Qwen2.5-1.5B）
- **sqlite3** `^5.1.7` —— 本地存储 + FTS 全文检索（不是向量库，是 FTS）
- **pdfjs-dist** `^6.0.227` / **pdf-parse** `^1.1.1` —— PDF 解析与预览
- **mammoth** `^1.8.0` / **word-extractor** `^1.0.4` / **docx** `^9.7.1` —— Word 解析与生成
- **marked** `^18.0.5` —— Markdown 渲染
- **keytar** `^7.9.0` —— API Key 安全存储于系统钥匙串
- **electron-store** `^8.1.0` —— 配置存储
- **本地模型：** Qwen2.5-1.5B-Instruct（Apache 2.0 开源协议）

> 一句话总结技术取舍：能用 SQLite 解决的，绝不引入第二个服务；能存在本地的，绝不上云。
